# Build Specification: Minimal Cross-Platform Markdown Block Editor

## 1. Project Name

Working name: **FocusMark**

The name is provisional. Keep the app name in configuration constants so it can be changed later without a large refactor.

## 2. Product Objective

Build a small, fast, minimal Markdown editor for Windows, macOS and Linux.

The core experience:

> The document normally appears as rendered Markdown. When the user edits a section, only the current Markdown block becomes raw editable Markdown. The rest of the document remains rendered.

The app should feel closer to **Notepad for Markdown** than to Obsidian, VS Code, Typora, Zettlr or a knowledge-base system.

Primary goals:

1. Fast launch.
2. Fast typing.
3. Minimal UI.
4. Small binary/install size.
5. Native file open/save behaviour.
6. Cross-platform support.
7. Open-source-friendly architecture.
8. Keyboard-first operation.
9. Configurable keyboard shortcuts.
10. Reliable fallback to full raw Markdown source mode.

Non-negotiable principles:

* Source Markdown is the single source of truth.
* Never silently reformat the user’s Markdown.
* Never corrupt files.
* Keep the app simple.
* Prefer predictable behaviour over clever behaviour.
* No telemetry.
* No account.
* No cloud sync.
* No database.
* No plugin marketplace in v1.

## 3. Target Platforms

Support:

* Windows 10/11 x64.
* macOS, Apple Silicon and Intel where practical.
* Mainstream Linux desktop distributions.

Package targets:

* Windows installer.
* macOS app bundle and/or DMG.
* Linux AppImage.
* Optional Linux deb/rpm later.

The app must be able to register itself as the default editor for:

* `.md`
* `.markdown`
* `.mdown`
* `.mkd`

When a user double-clicks a Markdown file, the app should open that file.

## 4. Technology Stack

Use:

* Tauri v2.
* TypeScript.
* Vite.
* Plain TypeScript or Svelte.
* CodeMirror 6.
* Rust only where required for Tauri backend/native integration.

Preferred frontend choice:

* Use **plain TypeScript** unless UI complexity justifies Svelte.
* Avoid React unless absolutely necessary.
* Avoid large UI frameworks.
* Avoid heavy component libraries.
* Avoid Monaco.
* Avoid Electron.
* Avoid server processes.

Markdown parsing/rendering:

* Use a CommonMark-compatible parser with GFM-style extensions.
* **Important Parse-Mapping Requirement:** Because the rendered block-editing model relies on matching exact character offsets, standard HTML-only parsers like `markdown-it` (which only output line ranges) are highly error-prone. The implementation must use a parser that constructs an Abstract Syntax Tree (AST) with precise character/byte offsets, such as **`remark`** (via `remark-parse` and the `mdast` node positions).
* Use **`DOMPurify`** as a lightweight sanitisation layer for rendered HTML to prevent XSS.

Tauri backend responsibilities:

* Native file open.
* Native file save.
* Native save-as.
* Native menus.
* File association handling.
* Open-with handling.
* Single-instance handling.
* Recent file persistence.
* Unsaved-change protection.
* Window title updates.
* Platform packaging.

Frontend responsibilities:

* Markdown rendering.
* Block mapping.
* Active block editing.
* Source mode editing.
* Keyboard shortcuts.
* Command registry.
* Formatting commands.
* Settings UI.
* Dirty state.
* Find.
* Status bar.
* Toolbar.
* Preferences.

## 5. Markdown Dialect

Default dialect:

* CommonMark-compatible Markdown core.
* Enable GitHub Flavored Markdown-style extensions:

  * Tables.
  * Task lists.
  * Strikethrough.
  * Fenced code blocks.
  * Autolinks.

Do not implement in v1:

* Mermaid diagrams.
* Math rendering.
* Footnotes unless the chosen parser supports them cheaply and safely.
* Wiki links.
* Backlinks.
* Tags database.
* Graph view.
* Embedded live web content.
* Plugin system.

The app must preserve the user’s source Markdown exactly except where the user edits it.

Do not auto-format.

Do not auto-renumber ordered lists except where explicitly required by direct user editing.

Do not normalise line endings for existing files. Preserve detected line endings.

## 6. Core UX Model

### 6.1 Default Mode: Rendered Block Editing

The default mode is **Rendered Editing Mode**.

In this mode:

* The document appears as rendered Markdown.
* Clicking or navigating to a rendered element activates the containing Markdown source block.
* Only that active block becomes raw editable Markdown.
* All other blocks remain rendered.
* The active block appears in-place.
* The active block uses a monospace font.
* The active block has subtle visual highlighting.
* Changes are written to the source document model.
* When the block loses focus, it returns to rendered view.

The app should feel like:

> A Markdown viewer where the current block instantly becomes Notepad.

### 6.2 Raw Source Mode

The app must also have **Raw Source Mode**.

In Raw Source Mode:

* The whole document is shown as raw Markdown.
* Editing is plain text editing.
* Use CodeMirror 6.
* It must be fast, reliable and boring.
* This mode is the fallback for unusual Markdown edge cases.

Switching between Rendered Editing Mode and Raw Source Mode must not alter the document.

### 6.3 Mode Toggle

The app must include a visible source toggle.

Available from:

* Minimal toolbar button.
* View menu.
* Configurable keyboard shortcut.

Default shortcut:

```text
Ctrl+E / Cmd+E
```

Toolbar label may be:

```text
Source
```

or an icon with accessible text.

When Source Mode is active, the toolbar button must show an active state.

## 7. Block Editing Behaviour

Markdown is block-structured, not purely line-structured. Therefore the editor should activate whole Markdown blocks rather than individual physical source lines.

### 7.1 General Block Rules

When a block is active:

* Replace the rendered block with a raw Markdown editor for that block.
* Use CodeMirror 6 or a lightweight editable text area for the active block.
* Prefer CodeMirror 6 for consistent editing, shortcuts and selection behaviour.
* Keep the editor visually in-place.
* Use monospace font.
* Preserve approximate vertical spacing to avoid excessive layout jumps.
* Commit changes live to `DocumentState.currentText`.
* **Typing Latency Optimization:** To prevent input lag and layout stuttering, the application must **not** perform a full AST re-parse or full document re-render on every single keystroke. Instead, update the active block's text locally, apply a running delta offset to any subsequent block position caches, and only trigger a full document AST parsing and rendering pass when the active block loses focus (`blur`) or after a generous debounce delay (e.g., 500ms to 1000ms of typing inactivity).
* Preserve cursor position as far as practical during re-rendering.
* Never alter unrelated blocks.

### 7.2 Headings

Inactive rendered heading:

```text
Project Notes
```

Active raw block:

```markdown
# Project Notes
```

A heading is edited as a single raw Markdown block.

### 7.3 Paragraphs

A paragraph is edited as the full raw paragraph block.

If the source paragraph spans multiple physical lines, all those lines become editable together.

Do not collapse, unwrap or rewrap paragraph lines automatically.

### 7.4 Lists

Lists should be edited as a block.

For v1, prefer activating the containing list block rather than individual list items.

Requirements:

* Preserve nested indentation.
* Preserve marker style.
* Pressing Enter at the end of a list item should create a new list item where practical.
* Pressing Enter on an empty list item should terminate the list where practical.
* Ordered lists should not be aggressively renumbered.
* User source formatting should be preserved.

### 7.5 Blockquotes

A blockquote should be edited as the whole quote block.

Example active block:

```markdown
> This is a quoted paragraph.
> It can span multiple source lines.
```

### 7.6 Fenced Code Blocks

A fenced code block should be edited as the whole fenced block.

Example active block:

````markdown
```powershell
Get-Process
```
````

Do not attempt per-line editing inside fenced code blocks in v1.

Syntax highlighting inside inactive rendered code blocks is optional and should be off by default for performance.

### 7.7 Tables

A Markdown table should be edited as the whole raw table block.

Example active block:

```markdown
| Name | Value |
|---|---:|
| A | 10 |
| B | 20 |
```

Do not implement spreadsheet-style cell editing in v1.

### 7.8 Horizontal Rules

A horizontal rule is edited as its raw source block:

```markdown
---
```

### 7.9 Images and Links

Images and links should render normally when inactive.

When the containing block is active, the user edits the raw Markdown source.

Do not implement modal link/image editors in v1.

### 7.10 HTML Blocks

Default v1 behaviour:

* Raw HTML should either be strictly sanitised or escaped.
* Do not execute scripts.
* Do not allow inline event handlers.
* Do not allow `javascript:` URLs.

If HTML block mapping is uncertain, activate the full HTML block as raw source.

## 8. Rendering Architecture

The source Markdown string is canonical.

Use this document state shape:

```ts
type DocumentState = {
  filePath: string | null;
  fileName: string;
  originalText: string;
  currentText: string;
  lineEnding: "\n" | "\r\n";
  hadBom: boolean;
  dirty: boolean;
  activeBlockId: string | null;
  selection: SourceSelection | null;
  lastSavedAt: number | null;
  fileModifiedAt: number | null;
  mode: "rendered" | "source";
};
```

Selection type:

```ts
type SourceSelection = {
  startOffset: number;
  endOffset: number;
};
```

Represent rendered Markdown as block models:

```ts
type MarkdownBlockType =
  | "heading"
  | "paragraph"
  | "list"
  | "blockquote"
  | "code_fence"
  | "table"
  | "thematic_break"
  | "html"
  | "image"
  | "unknown";

type RenderBlock = {
  id: string;
  type: MarkdownBlockType;
  startOffset: number;
  endOffset: number;
  startLine: number;
  endLine: number;
  source: string;
  html: string;
};
```

Rendering pipeline:

1. Parse Markdown source utilizing an AST parser (e.g. `remark-parse`) to capture accurate character offsets.
2. Produce block model with precise source ranges (`startOffset` and `endOffset`).
3. Render inactive blocks as DOM elements using sanitised HTML.
4. Render the active block as a raw CodeMirror 6 editor instance positioned in-place.
5. Keep a mapping from DOM elements to block IDs.
6. On click or keyboard navigation, set `activeBlockId`.
7. On edit, replace the source range for that block in `currentText`. During typing, offset calculations for subsequent blocks must be updated using a delta-tracker rather than triggering full document AST recalculations.
8. Recalculate block mapping after edits on block exit (`blur`) or upon a debounced delay.

For small documents, full re-render is acceptable.

For larger documents, optimise later with block-level rendering.

Performance thresholds:

* Under 100 KB: full re-render acceptable.
* 100 KB to 1 MB: prefer block-level re-rendering.
* Over 1 MB: Source Mode may be recommended if rendered editing becomes slow.

## 9. Editing Engine

Use CodeMirror 6 for:

* Raw Source Mode.
* Active block editor.

Active block editor requirements:

* No gutter.
* No minimap.
* No autocomplete.
* No heavy linting.
* No AI.
* No complex IDE features.
* Markdown syntax highlighting optional.
* Must support normal text editing.
* Must integrate with the central command registry.
* Must support formatting commands.

Undo/redo:

* **Unified State History:** Creating and destroying CodeMirror instances on each active block switch will clear local undo stacks. To maintain global undo/redo behavior, the application must orchestrate dynamic block edits through a single continuous background document model or persist individual block undo histories across editor transitions.
* Switching modes must not destroy the document state or its command history.

## 10. Formatting Commands

Formatting commands must work in both Rendered Editing Mode and Raw Source Mode.

They should operate on:

1. Current selection, if text is selected.
2. Current word, if no text is selected and the command makes sense.
3. Current block, for block-level commands.

*Formatting Context Note:* In Rendered Editing Mode, formatting commands function inside the active block's editor. If no block is currently active (focused), triggering a formatting command should automatically activate editing on the block containing the cursor focus point, or otherwise do nothing.

```text
Ctrl+B / Cmd+B
```

```markdown

```

```markdown
**selected text**
```


### 10.3 Inline Code

### 10.4 Strikethrough



### 10.9 Code Block

````markdown

````

## 13. Configurable Keyboard Shortcuts

The app must provide configurable keyboard shortcuts for as many commands as practical.

```ts

type ShortcutBinding = {
  key: string;
  ctrl?: boolean;
  shift?: boolean;
  alt?: boolean;
  meta?: boolean;
};
```

Requirements:

* Users can view all configurable actions.
* Users can search/filter actions.
* Users can assign a new shortcut.
* Users can clear a shortcut.
* Users can reset one shortcut to default.
* Users can reset all shortcuts to defaults.
* Detect conflicts.
* Warn before replacing an existing shortcut.
* Support platform-aware defaults:

  * Ctrl on Windows/Linux.
  * Cmd on macOS.
* Do not rely on shortcuts that the OS prevents the app from intercepting.
* Do not allow invalid shortcut combinations.
* Store shortcuts in preferences.
* Menu items should display the currently assigned shortcut.

Conflict example:

```text
Ctrl+B is already assigned to Bold.
Replace it?

[Replace] [Cancel]
```

If the user chooses Replace:

* Remove `Ctrl+B` from Bold.
* Assign `Ctrl+B` to the new command.

## 16. File Operations

### 16.1 New File

When creating a new file:

* Start with empty document.
* `filePath = null`.
* `dirty = false`.
* Default line ending:

  * Windows: CRLF.
  * macOS/Linux: LF.
* Window title:

```text
Untitled - FocusMark
```

### 16.2 Open File

Requirements:

* Use native file open dialog.
* Filter Markdown files:

  * `.md`
  * `.markdown`
  * `.mdown`
  * `.mkd`
  * all files fallback.
* Load as UTF-8.
* Detect UTF-8 BOM.
* Preserve BOM if present.
* Detect line endings.
* Preserve original line endings.
* Set window title to filename.
* Add path to recent files.
* If current document is dirty, prompt:

  * Save.
  * Discard.
  * Cancel.

### 16.3 Save

Requirements:

* If `filePath` exists, write to same path.
* Preserve line endings.
* Preserve BOM if originally present.
* Set `dirty = false`.
* Update `originalText`.
* Update `fileModifiedAt`.
* Update window title.

### 16.4 Save As

Requirements:

* Use native save dialog.
* Default extension: `.md`.
* Remember selected path.
* Save file.
* Update window title.
* Add path to recent files.

### 16.5 External File Changes

Simple v1 behaviour:

* If opened file changes on disk while document is dirty:

  * Warn before saving.
* If opened file changes on disk while document is not dirty:

  * Offer to reload.

Do not implement complex merge behaviour in v1.

## 18. File Association and Open-With Behaviour

The app must declare file associations in Tauri bundle configuration for:

* `.md`
* `.markdown`
* `.mdown`
* `.mkd`

Description:

```text
Markdown Document
```

MIME/content types where applicable:

```text
text/markdown
text/x-markdown
text/plain fallback only where necessary
```

When a Markdown file is opened from the OS:

* The app opens.
* The file path is passed to the app.
* The document loads automatically.

Single-instance behavior:
* Single-instance behavior must be enforced using the official `tauri-plugin-single-instance`.
* On Windows and Linux, the Rust backend must capture arguments from secondary launches via this plugin and emit an event containing the file path payload to the running frontend.
* On macOS, the backend must hook into native macOS URL/file-open notification hooks (e.g. `tauri::AppHandle` integration or event loop hooks) to capture "Open With" triggers while the app is active and forward the file path to the frontend.
* If the existing window is empty and clean, reuse it.
* If the existing window has a dirty document, prompt before replacing the current document.

Preferred v1:

* One document per window.
* Multiple windows allowed if practical.
* No tab bar in v1.

## 19. Unsaved Changes

When closing, quitting, opening another file, or creating a new file while dirty, prompt:

```text
Save changes to [filename]?

[Save] [Discard] [Cancel]
```

Behaviour:

* Save: save, then continue.
* Discard: continue without saving.
* Cancel: abort the action.

Must apply to:

* New file.
* Open file.
* Close window.
* Quit app.
* File launched from OS while current window has dirty changes.

## 23. Find

Implement simple in-document find.

## 25. Performance Requirements

Performance priorities:

1. Fast launch.
2. Low typing latency.
3. Minimal JS bundle.
4. Avoid heavy dependencies.
5. Avoid large runtime features.

Targets:

* Small Markdown files should open instantly.
* 100 KB Markdown files should feel immediate.
* Typing latency must be imperceptible.
* Full re-render acceptable under 100 KB.
* Optimise block-level rendering for larger files.
* Do not load syntax highlighters unless needed.
* Do not load Mermaid.
* Do not load MathJax.
* Do not load Monaco.
* Do not include a plugin framework in v1.
* Do not use a local database.

## 26. Security

Markdown rendering must be safe.

Requirements:

* Do not execute scripts from Markdown.
* Sanitise rendered HTML using a secure integration of DOMPurify.
* Block inline event handlers.
* Block `javascript:` URLs.
* External links open in the default browser.
* Remote images disabled by default.
* Local image rendering conservative.
* Tauri permissions should be least-privilege.
* Do not expose broad filesystem access to the frontend unless necessary.
* Prefer backend file commands for open/save.

## 27. Preferences Storage

Store preferences locally.

Requirements:

* Human-readable JSON is acceptable.
* Include schema version.
* Gracefully handle missing or corrupt preferences.
* Fall back to defaults if preferences cannot be read.
* Do not lose the user’s settings during normal app upgrades.

Suggested shape:

```json
{
  "version": 1,
  "preferences": {
    "theme": "system",
    "fontSize": 16,
    "editorFontFamily": "ui-monospace, SFMono-Regular, Consolas, Liberation Mono, monospace",
    "readerFontFamily": "system-ui, -apple-system, BlinkMacSystemFont, Segoe UI, sans-serif",
    "lineWidth": "normal",
    "showStatusBar": true,
    "showToolbar": true,
    "defaultMode": "rendered",
    "shortcuts": {}
  }
}
```

If `shortcuts` is empty, use defaults.

## 35. Quality Bar

The app is successful if it feels like:

> A Markdown viewer where the current block instantly becomes Notepad, with enough keyboard shortcuts that the mouse is optional.

It should remain:

* Small.
* Fast.
* Simple.
* Portable.
* Open source.
* File-based.
* Predictable.
* Keyboard-friendly.

It should not become:

* A full IDE.
* A note database.
* A project manager.
* A slow rich-text editor.
* A plugin platform.
* A complicated Markdown publishing system.

The core rule:

> Prefer simple, fast and predictable over clever.
